# v1 Evaluator Audit — Research Note

Scope: factual findings for ["Audit and correct the v1 evaluator"](https://github.com/JOHNNYMACONNY/mighty-mouse/issues/5). This is not an implementation or product decision.

## Findings

1. **The historical 15-task success measure is not a correctness measure.** Every task in [`data/evidence/tasks/`](../../data/evidence/tasks/) supplies a `test_script` with no assertions; each has a no-op requirement test and a no-op functional test (for example, [`task_001`](../../data/evidence/tasks/task_001_legacy_registry_ratelimiter.json)). The legacy evaluator treats a zero exit from that script as a passing test ([`tester.py`](../../src/mighty_mouse/services/verifiers/tester.py)); therefore all 15 recorded passes establish file materialization, scope/adherence checks, and a non-failing placeholder script—not behavior required by the task.

2. **The reported latency result is only historical/descriptive.** The recorded v1/Lean report shows 15/15 for both conditions and a 29.5% mean-success latency difference ([`validation_report.md`](../../data/evidence/results/validation_report.md)). The later bare control also passed 15/15, and the repository explicitly says that latency comparisons occur at different times and may not share identical runtime conditions ([`baseline_comparison.md`](../../data/evidence/results/baseline_comparison.md)). This creates a ceiling effect for success rate and prevents a contemporaneous causal latency comparison.

3. **The legacy result aggregation can collapse verifier detail.** The v1 validation runner reads only the first verifier entry from `logs/benchmark_results.json` and classifies every status other than literal `fail` as `SUCCESS` ([`run_validation.py`](../../eval/run_validation.py)). Its score scripts then reduce results to `status == "success"` ([`score_metrics.py`](../../eval/score_metrics.py), [`solve_benchmark.py`](../../eval/solve_benchmark.py)). A v2 evaluator needs a typed outcome schema and an explicit invalid/error state rather than implicit success-by-not-fail.

4. **v1 scope handling silently ignores AppleDouble and several broad artifact classes.** In isolated workspaces, the legacy scope verifier enumerates all files but counts basenames beginning `._` as ignored, along with every `.log`, `.tsv`, `.json`, and `.md` file ([`scope.py`](../../src/mighty_mouse/services/verifiers/scope.py)). Separately, its reset script deletes every `._*` file before a run ([`reset_workspace.sh`](../../eval/reset_workspace.sh)). Thus the historical scope outcome cannot distinguish an AppleDouble-free task workspace from one that was contaminated and then ignored or purged.

5. **AppleDouble was a real environment contaminant, not candidate behavior.** The build backend explains that `._*` sidecars on external macOS volumes can be mistaken for Python modules and stages sanitized release inputs ([`build_backend.py`](../../build_backend.py)); the v0.2.1 release records archive-level rejection of AppleDouble metadata ([`v0.2.1.md`](../releases/v0.2.1.md)). In the prospective paired study, the MM-001 and PORT-001 records say the evaluator applied the same neutral cleanup/staging procedure to both conditions and did not alter candidate source ([MM-001](../../data/evidence/real_project_trials/MM-001/README.md), [PORT-001](../../data/evidence/real_project_trials/PORT-001/README.md)).

6. **The repository already contains a prospective replacement protocol with the relevant validity controls.** It freezes task brief, base commit, allowed paths, acceptance commands, agent/model/settings, dependency snapshot, timeout, and environment before either condition; uses independent worktrees from one commit; retains raw transcripts/diffs/outputs; reruns acceptance in a fresh process; and invalidates rather than reconstructs a contaminated pair ([`real_project_study_plan.md`](../../data/evidence/real_project_study_plan.md)). The local-model study adds matched budgets/tools, randomized recorded order, normalized warm-up/cache state, held-out acceptance where applicable, a frozen corpus/model digest/seed/budget, and never silently reruns completed conditions ([`local-model-capability-study.md`](../local-model-capability-study.md)).

## Implication for a valid v2 rerun

The cited prospective protocol is the factual template: create a new, versioned evidence set from new frozen tasks and paired, isolated conditions; make acceptance executable and behavior-discriminating; record contamination detection and any neutral environment preparation for **both** conditions; and preserve task, model, environment, transcript, diff, command-output, and evidence-bundle hashes. Treat any changed prompt, runner, rubric, budget, task, or cleanup rule as a new protocol version. Do not overwrite, reclassify, pool, or rerun the historical v1 files: the repository already labels that corpus a synthetic latency experiment with permissive tests and excludes it from the new primary analysis ([`local-model-capability-study.md`](../local-model-capability-study.md)).
